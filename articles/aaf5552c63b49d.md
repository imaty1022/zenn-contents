## はじめに

妻と一緒にマタニティブログを運営しています。
（技術スタックはNuxt 3 + TypeScript + Tailwind CSS + microCMS で、Vercelにデプロイしています）

ブログの記事投稿はmicroCMSのGUIで妻が完結できるのですが、「デザインをちょっと変えたい」「このページに説明を追加したい」といったコードを伴う改修は、毎度私に依頼が来ていました。

これをどうにかしたい、というのが今回のモチベーションです。

## 目指したゴール

**非エンジニアの妻が、GitHubのIssueにコメントを書くだけでAIがコード修正してPRを出してくれる。私がレビューしてマージするだけでサイトに反映される。**

この形を実現することにしました。

## 最初に検討したアプローチ：OpenClaw + Slack連携

最初に思い浮かんだのは最近話題の **OpenClaw（×Claude Code）とSlackを連携させる** 方法でした。

Slackなら妻も普段から使っているし、「SlackのチャンネルにメッセージするだけでAIが動く」はUXとして理想的に見えました。

ただ、検討を進めるうちにいくつかの懸念が出てきました。

- SlackのBotセットアップ、Webhookの設定など、、、 **シンプルにめんどくさい**
- コードとの連携部分を自前で実装する必要がある、、、**シンプルにめんどくさい**
- 維持管理するコンポーネントが増える、、、**シンプルにめんどくさい**

**コンパクトさ・シンプルさを重視した結果、GitHub Actionsに絞ることにしました。**

GitHubのIssueは元々使っていますし（妻には使い方を教えないといけないけど）、`anthropics/claude-code-action` という公式Actionが存在し、設定もYAML数十行で完結します。余計なインフラを増やさずに済む、これが決め手でした。

## 構成

2つのワークフローを用意しました。

### 1. `claude.yml`：対話型（@claudeメンション）

IssueやPRのコメントに `@claude` と書くと反応します。妻がIssueを立てて「@claude このページのフォントを大きくして」と書けば、Claudeが修正してPRを出します。

```yaml
name: Claude Code

on:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]
  issues:
    types: [opened, assigned]

jobs:
  claude:
    if: |
      contains(github.event.comment.body, '@claude') ||
      contains(github.event.issue.body, '@claude')
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          github_token: ${{ secrets.GITHUB_TOKEN }}
```


### 2. `claude-team.yml`：パイプライン型（teamラベル）

複雑なタスク向け。Issueに `team` ラベルを付けると、まず **researcher** が調査してIssueにコメントし、続いて **implementer** がその調査結果を参照して実装・PRを作成します。

```yaml
name: Claude Agent Team (複雑タスク用)

on:
  issues:
    types: [labeled]

jobs:
  researcher:
    if: github.event.label.name == 'team'
    runs-on: ubuntu-latest
    permissions:
      contents: write
      issues: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - id: research
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          github_token: ${{ secrets.GITHUB_TOKEN }}
          prompt: |
            Issue #${{ github.event.issue.number }} の内容を読み、
            実装に必要なファイルと変更点を調査してください。
            調査結果をIssueにコメントしてください。

  implementer:
    needs: researcher
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          github_token: ${{ secrets.GITHUB_TOKEN }}
          prompt: |
            Issue #${{ github.event.issue.number }} を実装してください。
            Issueのコメントに調査結果があるので参照してください。
```
 
### レビューの強制

Claudeが直接mainにpushしないよう、2段構えで対策しています。

1. CLAUDE.md にルールを記載

ClaudeはリポジトリのCLAUDE.mdを必ず読むので、そこにルールを書きました。

```md
#### Git / GitHub ワークフロールール

- コードの変更を行う場合は、**必ず新しいブランチを作成**してから作業すること
- `main` ブランチへの直接 push は禁止
- 作業完了後は必ず **Pull Request を作成**し、レビューを依頼すること
```

2. GitHubのブランチ保護ルール

Settings → Branches → Branch protection rules で main ブランチに「Require a pull request before merging」を設定。技術的にもマージを強制できます。

### ハマったポイントと解決策

そのまま残しておきます。全部Claude Codeが解決してくれました。

#### ① id-token: write が必要

エラー：

```
Unable to get ACTIONS_ID_TOKEN_REQUEST_URL env variableCould not fetch an OIDC token. Did you remember to add `id-token: write` to your workflow permissions?
```

claude-code-action はGitHub App認証のためにOIDCトークンを使います。id-token: write が全jobのpermissionsに必要です。

#### ② prompt パラメータがコメント内容を上書きする

症状： Actions成功しているのにPRも変更もない。

原因： claude-code-action の prompt パラメータに値を設定すると、IssueやPRコメントの内容が無視され、prompt の文字列だけがタスクとして渡されます。

「PR作成を指示したい」という意図で prompt を追加したのが裏目に出ました。

```
# ❌ これをやるとコメント内容が無視されるprompt: |  必ずPRを作成してください。
```

解決策： prompt を削除し、CLAUDE.md にルールを書く。

#### ③ github_token を明示的に渡す必要がある

エラー：

```
error: Command failed: git fetch origin main --depth=1status: 128
```

github_token を省略すると、アクションはGitHub App経由のOIDCトークン取得を試みます。独自の設定なしではこれが失敗し、git操作が認証エラーになります。

```
# ✅ 明示的に渡すgithub_token: ${{ secrets.GITHUB_TOKEN }}
```

secrets.GITHUB_TOKEN はGitHub Actionsが自動発行するトークンなので、追加設定不要で確実に動きます。

#### ④ actions/checkout が必須

エラー：

```
fatal: not a git repository (or any of the parent directories): .git
```

GitHub Actionsのランナーはリポジトリを自動でチェックアウトしません。actions/checkout を明示的に追加して初めて .git が存在し、git操作が可能になります。

```
steps:
 - uses: actions/checkout@v4  # ← これが必須
 - uses: anthropics/claude-code-action@v1
```

### まとめ

Slack + OpenClawという選択肢も魅力的でしたが、既存のGitHubワークフローに乗っかることで新たなインフラを増やさず、最小限の設定で目的を達成できました。
エラーにかなり苦労しましたが、同じところでつまずく人の参考になれば嬉しいです。
